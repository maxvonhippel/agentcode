# token-rich-types — a language for building websites where the agent is the program

## Thesis

A web program is a tree of typed components. The *only* write operation is
**refinement**: provably narrowing a component while preserving its type
contract. Filling a TODO and prompting a component are the same primitive —
both produce a witness `t : T' <: T`, and the checker certifies the subtyping.

The agent that writes the program and the program are one object: components
refine *themselves* through a `prompt` primitive and interrogate themselves
through a `query` primitive, and the type system is at once (a) the
specification language, (b) the agent's navigation map, and (c) the guardrail
that keeps every self-edit type-safe. The agent may do anything as long as the
result type-checks as a subtype of the contract it was handed.

## The idioms that matter

1. REFINE IS THE ONLY WRITE.
   `?: T` is a hole — the top of T's refinement lattice. Both "fill this TODO"
   and `C.prompt(s)` produce `t : T' <: T`; the checker proves T' <: T. There is
   no other way to mutate the program, so the program can never break its own
   interface. `prompt` is one of two model-backed primitives; `query` is the
   other.

2. SPEC vs. WITNESS — the distinction that keeps idiom 1 honest.
   A component's *type* is its loose, abstract contract. The component is one
   *witness*. `prompt` narrows the witness: cheap, local, always a subtype, no
   re-proof. Removing something the contract *guarantees* is a different move —
   REVISING THE SPEC (widening the type) — which is explicit and re-discharges
   every dependent's subtyping obligation. Narrow freely; widen deliberately.

3. QUERY IS THE RECURSIVE READ.
   `C.query(q) -> A` interrogates a component's contents without mutating it —
   the read dual of prompt. It is recursive in the recursive-language-model
   sense: the tree is an environment to NAVIGATE, not a blob to paste. A query
   over the page decomposes into sub-queries over subtrees and aggregates, so
   only the relevant slice is ever read in full (no context rot). The token-rich
   type is the cheap index that decides which subtree to recurse into BEFORE
   reading it — typed routing replaces blind probing. `print` is this same idea
   in the visual modality; `query` is the semantic one. And `prompt` is built on
   `query`: it gathers context and locates its locus by querying, then refines.

4. TOKEN-RICH TYPES ARE THE MAP.
   Types carry enough structure — shape, layout, state, semantics — that the
   agent navigates *by type*. `page.prompt("add a banner")` routes to the child
   whose type is the right locus, with no human-specified path. Types are
   searchable (Hoogle-for-UI). The richness of the type *is* the navigation
   surface — this is what the name means.

5. COMPONENT TYPES ARE AN OPEN PRODUCT OF FACETS.
   A component's type is an intersection of independent facets — Shape & Layout &
   State today, Style & Behavior tomorrow. Each facet carries its own subtyping
   order, its own constraint theory (Layout: geometry; Style: contrast/spacing;
   Behavior: state-machine refinement), and its own fold into observation.
   Refinement narrows per facet. The kernel is facet-PARAMETRIC: it does not
   hardcode which facets exist, so Style (the CSS-equivalent) and Behavior (the
   JS-equivalent) slot in later as facets in the same homogeneous language —
   never as embedded sublanguages (idiom 7). The deterministic facets compile to
   WASM; the model-backed primitives are host imports — so a deployed page runs
   in the browser and can still refine itself against a model endpoint.

6. IDENTITY IS INTRINSIC.
   Every node has a stable short ID; `page.get("xy-2i-sk-1")` addresses it
   directly. The program is an ID-addressed graph of nodes, not a pile of files.
   A filesystem is optional sugar over the store.

7. ONE LANGUAGE, NO EMBEDDED SUBLANGUAGES.
   No `style="..."`, no CSS/JS-in-strings. Layout, style, and behavior are typed
   first-class values (idiom 5). The AST stays homogeneous and fully visible to
   the type system — there are no opaque string-holes the agent cannot reason
   about, and nothing the recursive `query`/`print` reads can be a black box.

8. OBSERVATION CLOSES THE LOOP.
   `print(dims [, state])` is a pure projection from a component to something
   perceivable — ASCII first (cheap, ideal for multimodal agents), image second.
   State is supplied, or left free and symbolically explored / fuzzed. Render is
   the feedback signal: prompt -> refine -> observe -> prompt.

9. CLICK IS A PRIMITIVE — THE SHARPIE.
   The renderer emits an id->region map, so a click resolves to its component in
   O(1). Right-click -> prompt box -> refine -> re-render is therefore built in,
   not bolted on. The interaction idiom this unlocks is new: you circle the bit
   you want changed (a click, an id) and SAY the change — a blend of prompting
   and coding where selection replaces navigation and utterance replaces
   editing. The same id->region map powers symbolic interaction.

10. DECIDABLE CORE + PROOF ESCAPE.
    Subtyping is decidable on a core fragment: structural subtyping plus
    SMT-backed refinements (e.g. "fits within dims", "no overflow"). When it is
    not decidable, you drop into the prover and supply a proof term by hand. The
    escape hatch speaks the same language the checker speaks.

11. LOCALITY OVER GLOBAL PLANNING.
    Use names before declaring them; an unresolved reference is just a hole in
    the import graph, discharged later. The agent never plans imports up front.

12. SELF-REFERENCE.
    `prompt` and `query` are methods on the value, backed by a model, fenced by
    the checker: a proposed refinement is accepted only if it checks as a subtype,
    else retried within a budget. The program carries the means of its own typed
    inspection and modification. Code and coder are one term.

13. HONEST INCAPACITY IS FIRST-CLASS (the scream).
    A prompted component answers with exactly one action: refine (typechecked),
    delegate (route down), escalate (push up), refuse (should not be done), or
    SCREAM — a structured declaration that the LANGUAGE ITSELF cannot express
    what was asked, with a diagnosis and a concrete extension proposal. Screams
    are recorded: the language's self-authored issue tracker, and the selection
    pressure for its own evolution (the endgame: the system benchmarks candidate
    language extensions born from its own screams).

14. THE PROGRAM CARRIES ITS OWN MEMORY.
    The store is a node: (store page journal). The enforcement machinery — and
    nothing else — appends every outcome to the journal as a typed entry.
    History has no surface form, so no refinement can fabricate or edit it, and
    it outlives what the page becomes. The journal is where the scream log
    lives, and where a future evolution loop reads its selection pressure.

## Roadmap (the parts that are still design, not code)

- PROMPT BUILT ON QUERY: replace paste-the-render context with query-gathered
  context once pages outgrow a single window. Done when a prompt on a large
  page reads only the slices it needs.
- TYPE-DIRECTED ROUTING: match an instruction against child types before asking
  a model. Done when page.prompt lands deep edits without a delegate call.
- MULTI-PAGE STORE: (store page… journal) with link navigation between roots.
  Done when :build grows a coherent multi-page site.
- STATE: input values and details open/closed as print(dims, state) parameters;
  symbolic/fuzzed exploration when unspecified.
- REGION MAP: the renderer emits id->region so a click IS the sharpie circle
  (the viewer currently selects via the DOM).
- PROOF ESCAPE: SMT for layout/refinement constraints ("fits within dims");
  manual proof terms where undecidable.
- FACETS: component types become an open product (Shape & Layout & State, then
  Style, then Behavior); kernel facet-parametric; adding a facet requires no
  kernel change. Style answers the standing rainbow scream.
- WASM: compile the deterministic kernel; prompt/query become host imports; a
  deployed page keeps refining itself in the browser.
- SELF-EVOLUTION: a loop that reads Screamed entries from the journal, proposes
  a vocabulary-table row, and benchmarks the extended language — the language
  maintains itself. Requires the vocabulary tables to live in the store.

Open questions: widening semantics (spec revision re-proves dependents); the
host seam protocol for WASM; state exploration policy (symbolic vs fuzz).
